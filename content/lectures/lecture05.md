# Lecture 05 — Инструменты тестировщика. Автоматизация vs Ручное тестирование

**Тема:** Обзор популярных инструментов для ручного и автоматизированного тестирования.
**Цель урока:** Познакомиться с современным инструментарием QA-инженера и понять, когда нужно автоматизировать тесты, а когда достаточно ручной проверки.
**Время:** ~20 мин.
**Ключевые понятия:** DevTools, Postman, Selenium, автотесты, ручное тестирование, чек-лист, тест-кейс.

### 1) Введение
Профессиональный тестировщик использует в работе множество инструментов. Они помогают не только искать баги, но и управлять процессом тестирования. Важно понимать разницу между ручным и автоматизированным тестированием и выбирать подходящий подход для каждой задачи.

### 2) Основная часть
**Инструменты ручного тестирования:**
*   **Баг-трекинговые системы:** Jira, Redmine, Trello (для управления задачами).
*   **TMS (Test Management System):** TestRail, Qase.io — для написания тест-кейсов и чек-листов, отчетности по результатам прогонов.
*   **Инструменты для работы с БД:** DBeaver, HeidiSQL — для проверки данных напрямую в базе.
*   **Charles / Fiddler:** Прокси-серверы для перехвата и изменения трафика между клиентом и сервером (незаменимы для тестирования мобильных приложений).
*   **Инструменты разработчика в браузере (DevTools):** Позволяют смотреть консоль ошибок, изучать сетевые запросы, эмулировать мобильные устройства.

**Инструменты автоматизации:**
*   **Postman / SoapUI:** Для автоматизации тестирования API (интерфейсов взаимодействия программ).
*   **Selenium WebDriver:** Самый популярный фреймворк для автоматизации тестирования веб-интерфейсов.
*   **Pytest / JUnit:** Фреймворки для написания автотестов на Python и Java соответственно.

**Ручное или автоматическое тестирование?**
*   **Ручное:** Используется для исследования (exploratory testing), проверки юзабилити, разовых задач, небольших проектов. Дешево на старте, требует времени при регрессе.
*   **Автоматическое:** Используется для рутинных проверок (регресс), нагрузочного тестирования, частого запуска одних и тех же тестов. Дорого в разработке, но окупается при долгой поддержке проекта.

### 3) Пример/практика
Представьте, что у вас есть сайт с формой входа.
*   Для **ручного** тестирования вы откроете сайт, введете логин и пароль и нажмете кнопку.
*   Для **автоматизации** вы напишете скрипт на Python + Selenium, который сам откроет браузер, введет данные и проверит, перешел ли пользователь на нужную страницу.
